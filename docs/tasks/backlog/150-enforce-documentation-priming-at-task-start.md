# Task 150: Enforce DOCUMENTATION.md and docs/ priming at task start

**Feature**: none
**Created**: 2026-04-08
**Depends on**: 149
**Blocks**: none

## Problem

5DayDocs has a single source of truth (`DOCUMENTATION.md`) and a curated reference tree (`docs/`). The whole system depends on AI agents and humans actually reading those before starting work. They mostly don't. Agents skim, humans skip, and both work from stale recall. Within a few sessions, the project drifts away from its own documented conventions, and the documentation itself starts to feel out of date because nobody is exercising it.

The fix is to make priming non-optional. When an agent or developer picks up a task, the act of picking it up should put `DOCUMENTATION.md` and the relevant `docs/` files in front of them, every time, automatically. Not as a polite suggestion in a CLAUDE.md somewhere, but as a structural property of the workflow.

There are two layers to this:

1. **Instruction layer.** The AI bootstrap files generated by task 149 tell the agent: "before doing anything on a task, read DOCUMENTATION.md and the docs/ files referenced in the task." This is necessary but not sufficient — it depends on the agent being well-behaved.
2. **Mechanism layer.** A `5day.sh task-start <id>` (or `5day.sh working <id>`) command that, when run, prints/loads/displays `DOCUMENTATION.md` and the task-relevant `docs/` files, so the priming happens *as a side effect of starting the task*. The agent doesn't have to remember; the command does it for them.

Both layers should ship together. The instruction layer reaches agents that don't run the command; the mechanism layer reaches agents (and humans) that do.

## Success criteria

- [ ] `./5day.sh task-start <id>` (or chosen verb) moves the task into `working/` and emits a priming bundle: `DOCUMENTATION.md` plus every `docs/` file the task's frontmatter references (Feature, related guides, etc.)
- [ ] Output format is suitable for both human reading and AI ingestion (clear section headers, file paths labeled)
- [ ] The command is idempotent and safe to re-run mid-task (re-prints the same bundle)
- [ ] AI bootstrap files generated by task 149 instruct the agent to run `./5day.sh task-start <id>` as the first action when picking up a task
- [ ] The instruction layer also has a fallback: "if the command isn't available, read these files manually" — listed by path
- [ ] Optional hook: a Claude Code `SessionStart` or `UserPromptSubmit` hook (configurable, off by default) that runs the priming command automatically when the agent enters a task context
- [ ] Documentation in DOCUMENTATION.md explains the priming model and why it exists (anti-drift rationale)
